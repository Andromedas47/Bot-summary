# MASTER.md — Financial Core Specification for Bot-summary
> เอกสารนี้เป็นข้อกำหนดทางการเงินหลัก (authoritative source) สำหรับ Bot-summary
> การแก้ code, database schema, Dashboard, report, export และ tests ในอนาคต ต้องไม่ขัดกับเอกสารนี้
> หาก code เดิมหรือเอกสารเดิมขัดกับเอกสารนี้ ให้ถือเอกสารนี้เป็นหลัก

---

## 1. Purpose and scope

เอกสารนี้กำหนดความหมายของยอดเงิน วิธีคำนวณ และคำแสดงผลสำหรับการสรุปรอบขายผลไม้ใน Bot-summary

ระบบต้องแยกยอดต่อไปนี้ออกจากกันอย่างชัดเจน:

1. **ยอดที่ต้องขายได้** — ยอดสุทธิจากสินค้า
2. **ยอดขายในใบ** — ยอดเงินที่รวมจากรายการการรับ/ใช้/ส่งเงินในใบขาว
3. **ส่วนต่างเงิน** — ผลต่างระหว่างยอดที่ต้องขายได้กับยอดขายในใบ
4. **ส่วนต่างสลิป** — ผลต่างระหว่างยอดเงินโอนในใบกับยอดสลิปที่ตรวจพบ

ห้ามใช้ยอดเหล่านี้แทนกัน

## 2. Definitions

### 2.1 หกหมวดสินค้า (Six produce categories)

ระบบรองรับ **หกหมวดสินค้า** ที่แยกออกจากกันอย่างชัดเจน:

| หมวด       | ความหมาย                                          |
| ---------- | ------------------------------------------------- |
| เบิก       | มูลค่าสินค้าที่ผู้ขายรับไปขายในรอบหลัก          |
| เบิกเพิ่ม  | มูลค่าสินค้าที่ผู้ขายรับเพิ่มในรอบเดียวกัน      |
| คืน        | มูลค่าสินค้าที่ผู้ขายนำกลับคืนในรอบหลัก        |
| คืนเพิ่ม   | มูลค่าสินค้าที่ผู้ขายนำกลับคืนเพิ่มในรอบเดียวกัน|
| คืนเสีย    | มูลค่าสินค้าที่เสียหายในรอบหลัก                 |
| คืนเสียเพิ่ม | มูลค่าสินค้าที่เสียหายเพิ่มในรอบเดียวกัน       |

หากไม่มีรายการในหมวดใด ให้ถือเป็น `0.00`

> **หมายเหตุ Legacy:** transaction_type "ชั่งคืนเพิ่ม" ในฐานข้อมูลปัจจุบันตรงกับหมวด **คืนเพิ่ม** ในโมเดลนี้

### 2.2 สามกลุ่มรวม (Three rollup groups)

```text
รวมเบิก    = เบิก + เบิกเพิ่ม
รวมคืน     = คืน + คืนเพิ่ม
รวมคืนเสีย = คืนเสีย + คืนเสียเพิ่ม
```

### 2.3 ยอดที่ต้องขายได้

ยอดสุทธิจากสินค้าที่ผู้ขายควรขายได้ หลังรวมเบิกเพิ่มและหักรายการคืน

```text
ยอดที่ต้องขายได้ = รวมเบิก - รวมคืน - รวมคืนเสีย
```

### 2.4 เงินโอน

ยอดเงินที่บันทึกว่าโอนเข้ามาในใบขาว ใช้สำหรับคำนวณยอดขายในใบ และใช้เทียบกับสลิปเงินโอน

### 2.5 ค่าใช้จ่าย

ยอดเงินขายที่นำไปใช้จ่ายตามที่บันทึกในใบ เช่น ค่าไฟ ค่าถุง ค่าทางด่วน หรือค่าใช้จ่ายอื่น

### 2.6 ส่งเงินสด

ยอดเงินสดก้อนที่ผู้ขายส่งตามที่ระบุในใบ

ในบางใบอาจแตกภายในเป็น:

```text
ส่งเงินสด
= ค่าแรง
+ เงินสดเหลือ
```

เมื่อมี `ส่งเงินสด` เป็นยอดรวมแล้ว ห้ามนำ `ค่าแรง` หรือ `เงินสดเหลือ` มาบวกซ้ำอีก
ค่าแรงและเงินสดเหลือมีไว้ตรวจสอบว่าแตกยอดส่งเงินสดถูกต้องหรือไม่ (display-only)

### 2.7 ยอดขายในใบ

ยอดเงินรวมจากรายการในใบขาว ใช้เป็นยอดที่นำไปเทียบกับยอดที่ต้องขายได้

```text
ยอดขายในใบ = เงินโอน + ค่าใช้จ่าย + ส่งเงินสด
```

`ยอดขายในใบ` ไม่รวมยอดที่ต้องขายได้ซ้ำเข้าไปอีก

### 2.8 ส่วนต่างเงิน

ผลต่างสำหรับตัดสินว่าส่งเงินขาด ส่งเงินครบ หรือส่งเงินเกิน

```text
ส่วนต่างเงิน = ยอดที่ต้องขายได้ - ยอดขายในใบ
```

### 2.9 ส่วนต่างสลิป

ผลต่างสำหรับตรวจว่าหลักฐานสลิปครอบคลุมยอดเงินโอนในใบหรือไม่

```text
ส่วนต่างสลิป = เงินโอนที่บันทึกในใบ - ยอดสลิปที่ตรวจแล้ว
```

## 3. Canonical formulas

### 3.1 กลุ่มรวมสินค้า

```text
รวมเบิก    = เบิก + เบิกเพิ่ม
รวมคืน     = คืน + คืนเพิ่ม
รวมคืนเสีย = คืนเสีย + คืนเสียเพิ่ม
```

### 3.2 ยอดที่ต้องขายได้

```text
ยอดที่ต้องขายได้ = รวมเบิก - รวมคืน - รวมคืนเสีย
```

### 3.3 ยอดขายในใบ

```text
ยอดขายในใบ = เงินโอน + ค่าใช้จ่าย + ส่งเงินสด
```

### 3.4 ส่วนต่างเงิน

```text
ส่วนต่างเงิน = ยอดที่ต้องขายได้ - ยอดขายในใบ
```

การแปลผล:

```text
ส่วนต่างเงิน = 0  → ส่งเงินครบ
ส่วนต่างเงิน > 0  → ส่งเงินขาด
ส่วนต่างเงิน < 0  → ส่งเงินเกิน
```

เวลานำเสนอส่งเงินเกิน ให้แสดงค่าสัมบูรณ์ เช่น:

```text
ส่วนต่างเงิน = -76.50
→ ส่งเงินเกิน 76.50 บาท
```

### 3.5 ส่วนต่างสลิป

```text
ส่วนต่างสลิป = เงินโอนที่บันทึกในใบ - ยอดสลิปที่ตรวจแล้ว
```

การแปลผล:

```text
ส่วนต่างสลิป = 0  → สลิปตรงยอดโอน
ส่วนต่างสลิป > 0  → ยอดโอนที่บันทึกมากกว่าสลิปที่พบ
ส่วนต่างสลิป < 0  → ยอดสลิปมากกว่ายอดโอนที่บันทึก
```

## 4. Source-of-truth hierarchy

### 4.1 ยอดสินค้า

ยอดในหกหมวด (เบิก, เบิกเพิ่ม, คืน, คืนเพิ่ม, คืนเสีย, คืนเสียเพิ่ม) ต้องมาจากข้อมูลรอบสินค้าและรายการสินค้าในระบบ

### 4.2 ยอดขายในใบ

ยอดขายในใบต้องมาจากข้อมูลที่บันทึกในใบขาว:

```text
เงินโอน + ค่าใช้จ่าย + ส่งเงินสด
```

ห้ามแทน `ส่งเงินสด` ด้วยการบวกค่าแรงและเงินสดเหลือเอง หากมีการกรอกยอดส่งเงินสดรวมอยู่แล้ว

### 4.3 เงินขาด/เกิน

เงินขาด/เกินต้องคำนวณจาก:

```text
ยอดที่ต้องขายได้ - ยอดขายในใบ
```

ห้ามใช้ยอดสลิปแทนยอดขายในใบ
ห้ามนำยอดรวมอื่นที่ไม่ได้อยู่ในสูตรมาแทนหรือบวกเพิ่ม

### 4.4 ข้อมูลไม่ครบ

**Null vs zero:** ค่า `null` หมายความว่า ยังไม่มีข้อมูล / ยังไม่ได้กรอก
ค่า `0` (ศูนย์) หมายความว่า มีข้อมูลแล้ว และค่าที่ยืนยันคือศูนย์
ห้ามแทน `null` ด้วย `0` เพราะจะทำให้ระบบเข้าใจผิดว่ากรอกครบแล้ว

หากยอดรวมใด ๆ ของเงินโอน ค่าใช้จ่าย หรือส่งเงินสดยังเป็น `null` ระบบต้องแสดงสถานะ:

```text
รอกรอกข้อมูลการเงิน
```

และต้องคืนค่า `null` สำหรับ ยอดขายในใบ, ส่วนต่างเงิน, และ displayAmount ทั้งหมด

ห้ามเดายอดจากข้อมูลย่อยที่ไม่ครบ
ห้ามสรุปเงินขาดหรือเงินเกินจากข้อมูลที่ยังไม่พร้อมตรวจ
ห้ามคำนวณส่วนต่างสลิปจากยอดสลิปที่ยังเป็น `null`

## 5. Status vocabulary and display wording

### 5.1 สถานะส่วนต่างเงิน

| เงื่อนไข            | สถานะที่แสดง           |
| ------------------- | ---------------------- |
| ข้อมูลการเงินไม่ครบ | รอกรอกข้อมูลการเงิน    |
| ส่วนต่างเงิน = 0    | ส่งเงินครบ             |
| ส่วนต่างเงิน > 0    | ส่งเงินขาด X.XX บาท    |
| ส่วนต่างเงิน < 0    | ส่งเงินเกิน X.XX บาท   |

### 5.2 สถานะส่วนต่างสลิป

| เงื่อนไข                      | สถานะที่แสดง                    |
| ----------------------------- | ------------------------------- |
| ยังไม่มีข้อมูลเงินโอน         | รอยอดเงินโอน                    |
| ยังไม่มีสลิปหรือยังตรวจไม่ครบ | รอตรวจสลิป                      |
| ส่วนต่างสลิป = 0              | สลิปตรงยอดโอน                   |
| ส่วนต่างสลิป > 0              | ยอดโอนมากกว่าสลิปที่พบ X.XX บาท |
| ส่วนต่างสลิป < 0              | ยอดสลิปมากกว่ายอดโอน X.XX บาท   |

## 6. Settlement reconciliation and slip reconciliation are separate

### 6.1 Settlement reconciliation

ใช้ตรวจเงินขาด/ครบ/เกิน:

```text
ยอดที่ต้องขายได้
เทียบกับ
ยอดขายในใบ
```

### 6.2 Slip reconciliation

ใช้ตรวจหลักฐานของเงินโอนเท่านั้น:

```text
เงินโอนที่บันทึกในใบ
เทียบกับ
ยอดสลิปที่ตรวจแล้ว
```

### 6.3 Non-interference rule

ผลตรวจสลิปห้ามเปลี่ยน:

- ยอดที่ต้องขายได้
- ยอดขายในใบ
- ส่งเงินขาด
- ส่งเงินครบ
- ส่งเงินเกิน

และผลส่งเงินขาด/ส่งเงินเกินห้ามถูกนำไปเปลี่ยนผลตรวจสลิป

## 7. Worked example

### 7.1 ยอดสินค้า (หกหมวด)

```text
เบิก          19,748.60
เบิกเพิ่ม          0.00
คืน           15,411.60
คืนเพิ่ม           0.00
คืนเสีย        2,093.50
คืนเสียเพิ่ม       0.00
--------------------------------
รวมเบิก       19,748.60
รวมคืน        15,411.60
รวมคืนเสีย     2,093.50
--------------------------------
ยอดที่ต้องขายได้  2,243.50 บาท
```

### 7.2 ข้อมูลในใบขาว

```text
เงินโอน        1,561.00
ค่าใช้จ่าย        60.00
ส่งเงินสด      2,320.00
  ├─ ค่าแรง       500.00  (display-only)
  └─ เงินสดเหลือ 1,820.00  (display-only)
```

### 7.3 ยอดขายในใบ

```text
ยอดขายในใบ
= 1,561.00
+ 60.00
+ 2,320.00
= 3,941.00 บาท
```

### 7.4 ส่วนต่างเงิน

```text
ส่วนต่างเงิน
= ยอดที่ต้องขายได้
- ยอดขายในใบ
= 2,243.50
- 3,941.00
= -1,697.50 บาท
```

ผลลัพธ์:

```text
ส่งเงินเกิน 1,697.50 บาท
```

### 7.5 ข้อควรระวังในตัวอย่าง

`ค่าแรง 500.00` และ `เงินสดเหลือ 1,820.00` เป็นรายละเอียดภายในของ `ส่งเงินสด 2,320.00`

ห้ามคำนวณแบบนี้:

```text
เงินโอน
+ ค่าใช้จ่าย
+ ส่งเงินสด
+ ค่าแรง        ← ผิด: นับซ้ำ
+ เงินสดเหลือ   ← ผิด: นับซ้ำ
```

เพราะจะนับยอดส่งเงินสดซ้ำ

## 8. Non-negotiable rules

1. ห้ามใช้ยอดที่ต้องขายได้แทนยอดขายในใบ
2. ห้ามใช้ยอดขายในใบแทนยอดที่ต้องขายได้
3. ห้ามนำยอดที่ต้องขายได้มาบวกเข้าในยอดขายในใบ
4. ห้ามนำค่าแรงและเงินสดเหลือมาบวกซ้ำ เมื่อมีส่งเงินสดยอดรวมแล้ว
5. ห้ามใช้ยอดสลิปแทนเงินโอนที่บันทึกในใบ
6. ห้ามให้ผลตรวจสลิปเปลี่ยนผลเงินขาด/ครบ/เกิน
7. ห้ามคำนวณเงินขาด/เกินจากยอดสลิป
8. ห้ามเดายอดส่งเงินสดจากรายการย่อย หากข้อมูลไม่ครบ
9. หากข้อมูลการเงินไม่ครบ ต้องแสดงรอกรอกข้อมูลการเงิน ไม่ใช่เดาผล
10. Code, Dashboard, report, export และ tests ต้องเรียกใช้ logic กลางชุดเดียวกันสำหรับสูตรทางการเงิน

## 9. Future implementation implications

### 9.1 Bot session data

ระบบ session ต้องเก็บหรือส่งต่อหมวดสินค้าอย่างน้อย (หกหมวด):

- เบิก
- เบิกเพิ่ม
- คืน
- คืนเพิ่ม
- คืนเสีย
- คืนเสียเพิ่ม

เพื่อคำนวณยอดที่ต้องขายได้อย่างตรวจสอบย้อนหลังได้

### 9.2 Dashboard fields

Dashboard ต้องแยก field ชัดเจน:

- เบิก
- เบิกเพิ่ม
- คืน
- คืนเพิ่ม
- คืนเสีย
- คืนเสียเพิ่ม
- รวมเบิก
- รวมคืน
- รวมคืนเสีย
- ยอดที่ต้องขายได้
- เงินโอน
- ค่าใช้จ่าย
- ส่งเงินสด
- ค่าแรง (รายละเอียดประกอบ display-only)
- เงินสดเหลือ (รายละเอียดประกอบ display-only)
- ยอดขายในใบ
- ส่วนต่างเงิน
- สถานะเงิน
- ยอดสลิปที่ตรวจแล้ว
- ส่วนต่างสลิป
- สถานะสลิป

### 9.3 Reconciliation report

รายงานต้องแสดงส่วน settlement และ slip แยกกันอย่างชัดเจน:

```text
Settlement:
  ยอดที่ต้องขายได้
  ยอดขายในใบ
  ส่วนต่างเงิน
  สถานะเงิน

Slip:
  เงินโอน
  ยอดสลิปที่ตรวจแล้ว
  ส่วนต่างสลิป
  สถานะสลิป
```

### 9.4 Export

PDF, Excel และ export ทุกชนิดต้องใช้ชื่อ field และสูตรเดียวกับเอกสารนี้
ห้าม export ชื่อเดิมที่ทำให้ `ยอดที่ต้องขายได้`, `ยอดขายในใบ`, และ `ส่วนต่างเงิน` ปะปนกัน

### 9.5 Tests

ต้องมี regression tests อย่างน้อยสำหรับ:

1. แต่ละหมวดใน หกหมวด ส่งผลต่อ rollup ที่ถูกต้อง
2. สูตรรวมกลุ่ม: รวมเบิก, รวมคืน, รวมคืนเสีย
3. ยอดที่ต้องขายได้ = รวมเบิก - รวมคืน - รวมคืนเสีย
4. ยอดขายในใบ = เงินโอน + ค่าใช้จ่าย + ส่งเงินสด
5. ค่าแรงและเงินสดเหลือไม่ถูกบวกซ้ำเมื่อส่งเงินสดมีอยู่แล้ว
6. เงินขาดเมื่อยอดที่ต้องขายได้มากกว่ายอดขายในใบ
7. เงินครบเมื่อยอดเท่ากัน
8. เงินเกินเมื่อยอดขายในใบมากกว่ายอดที่ต้องขายได้
9. ข้อมูลการเงินไม่ครบ (null) ต้องแสดงรอกรอกข้อมูลการเงิน
10. สลิปเปลี่ยนได้โดยไม่เปลี่ยนสถานะเงิน (slip/money non-interference)
11. สถานะเงินเปลี่ยนได้โดยไม่เปลี่ยนผลตรวจสลิป

13. Legacy mapping: "ชั่งคืนเพิ่ม" → หมวดคืนเพิ่ม

## 10. Change-control rule

ก่อนแก้สูตร, schema, Dashboard, report, export หรือ tests ที่เกี่ยวข้องกับการเงิน ต้องตรวจเอกสารนี้ก่อนเสมอ

หากต้องเปลี่ยนความหมายของ field หรือสูตร:

1. แก้เอกสารนี้ก่อน
2. ระบุเหตุผลและผลกระทบ
3. review ให้ชัดเจน
4. จึงค่อยเปลี่ยน code และ database
